# Ch.3　データを「学ばせる」技術

## 今日の問い：機械が自動でルールを見つけるとはどういうことか

### ─── 前半（昼食前）───

---

## 機械学習とは何か ── ルールを「書く」から「学ばせる」へ

```
【従来のルールベースアプローチ】

 人間がルールを考える
        ↓
 「アルコール > 13.5 かつ プロリン > 900 なら wine_0」
 「アルコール < 12.5 なら wine_1」
 「それ以外は wine_2」
        ↓
 変数が増えるほどルールが複雑になり、人手では限界が来る
```

```
【機械学習のアプローチ】

 データ（特徴量 X）と正解（y）を大量に見せる
        ↓
 「このパターンならこのクラス」というルールを自動で見つける
        ↓
 人間はルールの設計ではなく「データ準備と結果の解釈」に集中できる
```

> 📌 **機械学習は「答えを教えながら勉強させる家庭教師」のようなものです。**
> 大量の例題（データ）から、機械が自力でパターンを学びます。

---

## 分類問題とは何か ── 「どのカテゴリか」を予測する

```
【分類問題 ＝ 出力（予測結果）がカテゴリになる問題】

  入力（特徴量 X）          予測（y）
  ──────────────           ──────────────────
  化学成分 13列    →       wine_0 / wine_1 / wine_2
  メールの文章     →       迷惑メール / 通常メール
  細胞の形・大きさ  →       良性 / 悪性
  製品の測定値     →       良品 / 不良品
```

**分類と回帰の違い（Ch.4 の予告）**

| | 分類（Ch.3） | 回帰（Ch.4） |
|--|------------|------------|
| **予測するもの** | カテゴリ（どのクラスか） | 数値（どのくらいか） |
| **例** | 品種が wine_0 か wine_1 か | 次の時刻のセンサー値は何か |
| **評価指標** | 正解率・混同行列 | R²・MAE（誤差） |

> 📌 **今日の演習は「3品種のどれか」を予測する分類問題です。**

---

## 特徴量（X）と正解ラベル（y）の役割

```
【X（特徴量）と y（正解ラベル）を分ける】

         X（手がかり・入力）                    y（答え・出力）
  ┌────────────────────────────────┐       ┌──────────┐
  │ alcohol  proline  flavanoids  │       │  wine_0  │
  │   14.2    1065      2.8       │  →    │  wine_0  │
  │   13.2    1050      2.5       │       │  wine_1  │
  └────────────────────────────────┘       └──────────┘
       モデルへの「入力」（13列）             モデルが予測する「答え」
```

```
⛔ 重要なルール：y（品種名）は X に含めてはいけない

なぜ？
  → 「答えを見ながら学習させる」＝ カンニングと同じ
  → 答えを知っている問題しか正解できない
  → 実務では「これから手に入るデータ」を予測するため、
    答えが含まれていることはありえない
```

> 📌 **「X ＝ 手がかり、y ＝ 正解」。この分け方が教師あり学習の出発点です。**

---

## データを2つに分ける理由 ── 「本番試験」の考え方

```
【全データで学習・全データでテストすると何が起きるか】

  全データ（178件）で学習
          ↓
  同じ178件でテスト
          ↓
  「見たことがある問題を解く」状態（丸暗記）
          ↓
  正解率 99%！でも…
  実務の新しいデータを渡したら → 60% しか正解できない（使い物にならない）
```

```
【正しいデータの使い方（Hold-out法）】

  全データ（178件）
     │
     ├── 訓練データ（80% ＝ 142件）  → モデルが「勉強する」データ
     └── テストデータ（20% ＝  36件）  → 学習後に「1回だけ採点する」データ
                                         ↑
                                 本番試験なので学習には絶対使わない
```

> 📌 **「訓練で高精度・テストで低精度」＝ 過学習。**
> 暗記した人が初めて見る問題で失敗するのと同じです。

---

<!-- _class: supplement -->

## 📎 補足｜3分割（Train・Validation・Test）の考え方

```
【実務では「3つ」に分けることが多い】

  全データ
     │
     ├── 訓練データ（Train）     → モデルのパラメータを学習させる
     ├── 検証データ（Validation）→ ハイパーパラメータを調整する
     └── テストデータ（Test）    → 最終的な性能を評価する（1回だけ）
```

**それぞれの役割**

| データ | 役割 | 例え |
|--------|------|------|
| **Train** | 「勉強する」 | 教科書で問題を解く |
| **Validation** | 「模擬試験を受けて勉強方法を調整する」 | 模擬テストで弱点を見つける |
| **Test** | 「本番試験を受ける（1回だけ）」 | 入試本番 |

> 📌 **今日の演習では Train と Test の2分割で進めます。**
> 実務でモデルを精緻化する際は、検証データを使った3分割が標準的です。

---

## 過学習とは何か ── 「丸暗記」の罠

```
【過学習（Overfitting）のイメージ】

訓練データ（142件）：正解率 99%
                      ↑ 全パターンを「暗記」している状態
テストデータ（ 36件）：正解率 65%
                      ↑ 初めて見るデータには対応できない
```

```
【過学習が起きやすい条件】

  ① モデルが複雑すぎる（パラメータが多すぎる）
     → 訓練データのノイズまで学んでしまう

  ② 訓練データが少なすぎる
     → 少ない例から無理やりルールを作ってしまう

  ③ 訓練データとテストデータの分布が大きく違う
     → 学んだパターンが実務では通用しない
```

**過学習を防ぐ主な方法**

| 方法 | 内容 |
|------|------|
| **データを増やす** | 学習に使えるデータ量を増やす |
| **モデルを単純にする** | パラメータを絞る・制約をかける |
| **正則化** | モデルが複雑になりすぎないよう罰則をかける（Ch.4 で詳述） |

> 📌 **過学習は「試験対策しかしていない人」が本番で失敗するのと同じ構造です。**

## 決定木の仕組み ── Yes/No の問いを重ねて絞り込む

```
【決定木 ＝ 「Yes / No の質問」を繰り返してカテゴリを特定する】

          アルコール度数 > 13.0 ？
                /              \
             Yes                 No
              ↓                   ↓
      プロリン > 900 ？       → wine_1 / wine_2 の候補
          /       \
        Yes         No
         ↓           ↓
       wine_0       wine_1
```

**決定木の特徴**

| 長所 | 短所 |
|------|------|
| なぜその予測をしたか「理由」が説明しやすい | 木が深くなると訓練データを丸暗記しやすい（過学習） |
| 視覚的に理解しやすい | データが少し変わると木の形が大きく変わる |

> 📌 **「なぜ wine_0 と判定したか」を人に説明しやすいのが決定木の最大の強みです。**
> しかし単独では精度が安定しにくいという弱点があります。

---

## ランダムフォレストとは ── 多数決で精度を安定させる

```
【ランダムフォレスト ＝ 決定木を「100本作って多数決」する】

訓練データの一部をランダムに選んで決定木を作る（100回繰り返す）
      │
  ┌───┼───┐
  木①  木②  木③  …（計100本）
wine_0 wine_1 wine_0
      │
   多数決 → wine_0（2票 vs 1票）← 最終予測
```

**なぜ1本より100本の方が良いのか**

```
  1本の決定木：
    特定の訓練データに依存してしまう → 過学習しやすい

  100本の決定木の多数決：
    各木が異なるデータの「一部」を見ている
    → 1本が外れても他の99本でカバーできる
    → 安定した精度が出やすい（過学習しにくい）
```

> 📌 **今日の演習で使うのが このランダムフォレストです。**
> 特徴量重要度（どの変数が分類に貢献したか）も自動で教えてくれます。

## 📎 補足｜分類アルゴリズムの系譜 ── ポジションマップ

```
【主な分類アルゴリズムの位置づけ】

精度
高│                              ★ XGBoost
  │                  ★ ランダムフォレスト  ← 今日使うのはここ
  │          ★ 決定木
  │  ★ ロジスティック回帰
  │  ★ SVM
低└──────────────────────────────────────── 解釈しやすさ
```

| **ランダムフォレスト** | 高精度・過学習しにくい | 精度優先の実務タスク |

## 正解率（Accuracy）の限界 ── 「全部正常」でも高くなる問題

```
【例：工場の不良品検知（不均衡データ）】

  全データ 1000件
  ├── 正常品：950件（95%）
  └── 不良品：  50件（ 5%）

  モデル A：「全部正常」と予測するだけ
  → 正解率 ＝ 95%
  → しかし不良品 50件を全部見逃している（実務では最悪の結果）

  モデル B：正解率 90%、しかし不良品 45件を検知している
  → 実務ではモデル B の方がはるかに価値が高い
```

> 📌 **正解率は「全体の成績」にすぎません。**
> 「何を正解したか・何を間違えたか」は混同行列を見ないとわかりません。

---

## 混同行列で「何を間違えたか」を見る

```
【混同行列 ＝ クラスごとの正解・不正解を一覧する表】

               予測：wine_0  予測：wine_1  予測：wine_2
実際：wine_0  [  ✅ 13   ] [   ❌  0   ] [   ❌  0   ]
実際：wine_1  [  ❌  0   ] [  ✅ 20   ] [   ❌  1   ] ← 1件を wine_2 と誤判定
実際：wine_2  [  ❌  0   ] [   ❌  1   ] [  ✅ 12   ] ← 1件を wine_1 と誤判定

対角線（✅）＝ 正解     対角線以外（❌）＝ 間違い
```

**混同行列の4つの領域（2クラスの場合）**

| | 予測：陽性（不良） | 予測：陰性（正常） |
|--|-----------------|-----------------|
| **実際：陽性（不良）** | TP（真陽性）正しく不良と検知 | FN（偽陰性）不良を見逃した |
| **実際：陰性（正常）** | FP（偽陽性）正常を誤って不良と判定 | TN（真陰性）正しく正常と判定 |

> 📌 **正解率が高くても「特定クラスを全部見逃している」可能性があります。**
> 必ず混同行列で確認することが重要です。

## Precision・Recall・F1 の意味 ── 何を重視するかで指標を選ぶ

```
【Precision（適合率）── 誤報を減らしたい場面で使う】

「不良と予測した中で、本当に不良だった割合」

  例）100件を「不良」と判定 → 実際に不良だったのは 80件
      Precision ＝ 80 ÷ 100 ＝ 80%

  重視する場面：誤った判定のコストが高い（調査・停止コストがかかる）

【Recall（再現率）── 見逃しを防ぎたい場面で使う】

「本当の不良を、どれだけ検知できたか」

  例）実際の不良 100件中、90件を検知できた
      Recall ＝ 90 ÷ 100 ＝ 90%

  重視する場面：見逃しのコストが高い（医療・安全・リコール）

【F1 Score ── PrecisionとRecallのバランスをとる】

  どちらも大切な場面で使う調和平均
  どちらかが低いと F1 も低くなる仕組み
```

> 📌 **「見逃しコスト」と「誤報コスト」のどちらが大きいかで指標を選びます。**

> 💡 **考え方のヒント：**
> ・「不良品を良品として出荷してしまう」→ **Recall を優先**（見逃しが致命的）
> ・「良品を不良品として廃棄してしまう」→ **Precision を優先**（誤報のコストが大）
> ・「悪性を良性と診断してしまう」→ **Recall を優先**（見逃しが命取り）
>
> Precision を上げると Recall が下がり、Recall を上げると Precision が下がります。
> **「どちらをどこまで許容するか」はビジネスの判断です。**
