## コースのゴール（一言）

> **「自分の業務でデータ分析を試してみたい」と思える状態にする**

コードを覚えさせるのではなく、「データサイエンティストとしてこういうことができる」という実務イメージを体験させる。

---

## 設計の3大原則

```
① 実務アウトプット起点
   各チャプターの終わりに「実務で使えるもの」を手元に残す
   → ただの操作体験で終わらせない

② 認知負荷の最小化
   わからないコードが増えると初心者は脱落する
   → コードは削ぎ落とす。座学スライドはコード説明ゼロ

③ スライド先行設計
   ノートブックを書く前に「何を教えるか・どこで発問するか」をPowerPointで確定
   → slides/ → lectures/ → notebooks/ の順で作る
```

| **実務での対応物** | 工場設備の過熱検知レポート・センサー異常アラートの原型 |

## 座学30分の共通フォーマット（全チャプター）

> **座学はスライド（Marp .md）のみ。** コードは扱わない。

```

```

## 演習40分の共通フォーマット（全チャプター）


## 生成AI（Copilot）活用方針

### 使い方：Copilot Chat（ウェブブラウザ）のみ

> インライン補完（Tab キー）は使わない。**ブラウザで開いた Copilot Chat への質問に統一する。**  
> `https://copilot.microsoft.com` または `https://github.com/copilot`  
> 推奨レイアウト：画面左半分 = JupyterLab、画面右半分 = ブラウザ（Copilot Chat）

### 演習のコードはすべて Copilot に書いてもらう

演習コードは Copilot に生成してもらうことを推奨。  
受講生の仕事は「**何をしたいかを言語化してプロンプトを書く**」ことと「**結果を解釈すること**」。

### 質問の型（全演習共通）

```
【やりたいこと】〇〇したい
【使うライブラリ】pandas, matplotlib など
【データの形】DataFrame で列名は〇〇、行数は〇〇件
【環境】Python 3.8.6、Windows、JupyterLab
【わからない点】〇〇の書き方 / 〇〇でエラーが出た
```

### Copilot 活用ルール

| タイミング | ルール | 理由 |
|-----------|--------|------|
| 座学30分中 | **使用禁止** | 概念を自分の頭に入れることが先 |
| 演習 問1〜2 | **テンプレートで質問する** | 「何をしたいか」を言語化する練習 |
| 演習 問3（考察） | コードはOK・**考察文は自分で書く** | 解釈はDSの本来の仕事 |
| 演習 問4（発展） | **自由に活用** | プロンプトの工夫も練習 |

> 詳細テンプレート集 → `redesign/07_copilot_prompt_guide.md`

## スライド（座学）の設計方針

座学スライドでは **演習の操作説明はしない**。  
代わりに「その演習を理解するために必要なデータサイエンスの周辺知識・概念」を教える。

```
座学スライドで教えること（例）
  Ch.1: DataFrameとは何か / なぜpandasを使うのか / 欠損値が生じる理由
  Ch.2: EDAとは何か / グラフの選び方 / 相関 ≠ 因果
  Ch.3: 異常検知とは / 正常を「定義する」とはどういうことか / Zスコアとは
  Ch.4: なぜデータを分割するか / スケーリングの必要性 / 混同行列の読み方

座学スライドで教えないこと
  × 演習の手順説明
  × コードの構文解説（→ 演習ノートブック + Copilot に任せる）
```
