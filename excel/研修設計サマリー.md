# Excel VBA 研修 2日目 ── 研修設計サマリー

> このドキュメントは、研修の「なぜ・何を・どう教えるか」を講師・企画者が共有するためのものです。

---

## 1. コアメッセージ

```
VBA は「コードを書く技術」ではなく、
「繰り返しの作業から解放され、本当に価値のある仕事に時間を使う」ための道具である。
```

Day 1 で「動くもの」を作った受講生が、Day 2 では「職場で実際に使えるツール」に仕上げることができる──  
この一点が、2日間研修全体の到達目標です。

---

## 2. 対象受講生の前提

| 項目 | 内容 |
|---|---|
| 経験レベル | 初学者〜初級（VBA を触ったことがあるが自力で書けない） |
| Day 1 終了時点のスキル | Sub / If / For / Do While / Workbooks.Open / Dir の基礎操作 |
| 業務背景 | Excel を日常的に使用している。定型作業の自動化ニーズがある |
| モチベーション | 「便利そう」と感じているが、業務で実際に使えるイメージが持てていない |

## 4. 各ブロックの設計意図

### Block 1｜デバッグ入門

**伝えたいこと**
> エラーは「失敗」ではなく「ヒント」である。適切な道具を使えば、誰でも冷静にバグを探せる。

**なぜここから始めるか**
- Day 1 の演習でバグに直面した受講生が「何をすればよいかわからなかった」という経験を持っている
- 「コードが動かない = 諦める」のループを断ち切ることが、自律的な学習の前提になる
- デバッグスキルは後続ブロックの演習でも即座に役立つ

**学びのゴール**
- エラーの3種類（コンパイル・実行時・論理）を区別できる
- Debug.Print、ブレークポイント（F9）、ステップ実行（F8）を使い分けられる
- 「合計が合わない」など、実務でよくある症状に対処できる

**演習の設計**
- 意図的にバグを2箇所仕込んだコードを渡す
- バグの種類：①ループ開始行のずれ（ヘッダー処理）②列番号のずれ
- どちらも実務で頻出する凡ミスであることを体感させる
- Debug.Print のコメントアウトを外すだけで手がかりが得られる設計

---

### Block 2｜Function（コードの再利用）

**伝えたいこと**
> 同じ処理を1か所にまとめることで、コードは「修正しやすく・読みやすく・使い回せる」ものになる。

**なぜここか**
- Day 1 時点のコードは「動いているが、読みにくく、修正が大変」な状態
- Function の概念は「コピペをなくす」という直感的な動機から入れる
- 「最終行取得」は全ての演習・業務マクロで必要になる汎用パーツ

**学びのゴール**
- Sub と Function の役割の違いを説明できる（戻り値の有無）
- 引数・戻り値の書き方を理解している
- 自分のマクロの中から「Function にできる処理」を見つけられる

**演習の設計**
- 3ステップ構成：Function を単体で作る → 別の Function を作る → Sub から呼び出す
- Step 1 完了後に `Debug.Print 売上評価(1200000)` で単体テストできる設計
- Before/After の比較で「Function を使う前後」の違いを視覚的に理解させる

---

### Block 3｜シートの操作

**伝えたいこと**
> 「毎月同じ作業」は、1日目に学んだ If と Dir だけで自動化できる。新しい文法は不要。

**なぜここか**
- シート操作は実務頻度が最も高い（月次レポート・振り分け・集計）
- 「シートをループで作る」「月ごとに振り分ける」は Day 1 の知識の延長で書ける
- 「エラーを防ぐ」思想を具体的なコードパターンで体感させる

**学びのゴール**
- シートを変数に入れて操作できる（`Set ws = Worksheets("集計")`）
- 動的にシートを作成・削除できる
- `Dir()` でファイルを「開く前に確認する」パターンが書ける

**演習の設計**
- Step 1：1〜12月のシートを自動生成（存在する場合は削除して再作成）
- Step 2：全データを Month() 関数で月ごとに振り分け
- 「月次報告書を毎月手動でシート追加している」という実態と直結させる

---

### 総合演習｜支店別売上集計ツール

**伝えたいこと**
> 2日間で学んだことを組み合わせると、実務で使えるツールが作れる。

**演習の設計**
- 4支店 × 6ヶ月 = 24ファイルを自動集計するツールを完成させる
- テンプレートに TODO が3箇所（必須2・推奨1）
  - TODO-1：`If Dir()` でファイルの存在確認（Block 3 の知識）
  - TODO-2：`最終行取得` Function を呼び出す（Block 2 の知識）
  - TODO-3：`ScreenUpdating` 等の最適化（Block 3 の知識）
- 「動けば完成」ではなく、ステップごとに動作確認して仕上げていく設計

## 5. 発問の設計方針

各ブロックの座学冒頭に「発問」を置く意図：

| 発問のタイミング | 目的 |
|---|---|
| 座学の冒頭 | 受講生の経験と知識を結びつけ、「自分ごと」として聞かせる |
| 業務落とし込みの末尾 | 抽象的な知識を具体的な行動に変換させる |
| 総合演習後のアクションプラン | 「明日何をするか」を宣言させ、定着率を上げる |

**発問の例**

- Block 1 導入：「マクロが動かなかったとき、これまでどう対処していましたか？」
- Block 2 導入：「同じ処理を何度もコピペして書いたことはありますか？」
- Block 3 導入：「毎月手動でシートを追加している作業はありますか？」

---

## 6. 「業務への落とし込み」の設計意図

各ブロックの最後に 5〜10 分の「業務への落とし込み」を設ける理由：

1. **自己効力感の醸成**：「自分の業務でも使えそう」という実感が、継続学習の動機になる
2. **転移の促進**：ケーススタディと自分の業務との橋渡しをしないと、知識が孤立する
3. **アウトプット学習**：「書いて宣言する」ことで記憶の定着率が上がる

> この時間を「省略してよい時間」と捉えないこと。  
> ここが受講生の「自信」につながる最重要の場面です。
